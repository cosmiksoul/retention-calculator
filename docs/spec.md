## 1. Общее описание

Веб-калькулятор для прогнозирования retention-кривой и расчёта LTV на основе исторических данных когорт.

**Целевая аудитория:** широкий круг продуктовиков, маркетологов и менеджеров. Инструмент должен быть понятен человеку без аналитического бэкграунда (с разумными дефолтами и подсказками), но при этом давать точные цифры профильному аналитику.

**Не black box.** Принципиально: пользователь должен понимать, по какой формуле строится прогноз и откуда взяты бенчмарки. Поэтому:
- Раздел «Methodology» — полноценная страница, не tooltip и не collapsible-блок
- Каждый пресет показывает badge качества данных (`robust` / `moderate` / `estimated`) и ведёт на свою секцию методологии

**Формат:** Single-page web app
**Хостинг:** GitHub Pages (жёсткое требование — статика, без бэкенда)
**Репозиторий:** `cosmiksoul/retention-calculator` (рабочее название)

## 2. Целевая аудитория и сценарии использования

### Кто будет пользоваться
- Продуктовые аналитики, которые хотят быстро прикинуть LTV без настройки Python/R
- Маркетологи, которым нужно оценить окупаемость привлечения
- Менеджеры и фаундеры, которым нужна визуальная оценка «когда отобьётся»
- Люди, готовящиеся к собеседованиям — как шпаргалка и демо-инструмент

### Основные сценарии
1. **Быстрая прикидка.** Ввёл 3-5 точек retention + ARPU → получил график и LTV
2. **Сравнение с бенчмарками.** Выбрал пресет индустрии → увидел типичную кривую → ввёл свои данные → сравнил
3. **Презентация стейкхолдерам.** Экспорт графика/таблицы для вставки в презентацию или отчёт
4. **Обучение.** Поиграть с параметрами, понять как retention влияет на LTV, где точка breakeven

## 3. Функциональные требования

### 3.1. Входные параметры

#### Режимы ввода данных

Три режима на выбор (табы или radio-переключатель):

**Режим 1: Manual Input (по умолчанию)**

Пользователь вводит retention-точки вручную.

| Параметр | Тип | Дефолт | Описание |
|----------|-----|--------|----------|
| Retention points | Массив пар (day, %) | D1: 40%, D7: 20%, D14: 15%, D30: 10% | Минимум 3 точки, максимум 10. Пользователь может добавлять/удалять точки |
| Cohort size | Number input | 1000 | Начальный размер когорты |
| Period unit | Select | Days | Days / Weeks / Months |

**Валидация:**
- Retention % должен быть от 0 до 100
- Day/period должен быть > 0
- Точки должны быть в хронологическом порядке
- Retention не может расти (каждая следующая точка ≤ предыдущей)
- Минимум 3 точки для фиттинга

---

**Режим 2: Paste Cohort Table (когортный треугольник)**

Пользователь вставляет когортную таблицу из аналитического инструмента (Amplitude, Mixpanel, GA4, SQL-выгрузка).

**Формат ввода:** textarea для вставки таблицы. Поддерживаемые форматы:
- TSV (копипаст из Google Sheets / Excel)
- CSV

**Ожидаемая структура:**

```
Cohort      | D0    | D1   | D7   | D14  | D30
2026-04-01  | 1200  | 480  | 240  | 168  | 108
2026-04-08  | 980   | 412  | 196  | 137  | —
2026-04-15  | 1100  | 462  | 220  | —    | —
2026-04-22  | 1050  | 399  | —    | —    | —
```

**Логика обработки:**
1. Парсинг таблицы, определение разделителя (tab/comma/semicolon)
2. Определение типа данных: абсолютные числа или проценты (эвристика: если значения в D1+ < D0 и > 1 → абсолютные; если ≤ 1 или содержат % → проценты)
3. Конвертация абсолютных чисел в % от D0 для каждой когорты
4. Усреднение retention % по всем когортам для каждого периода (используются только заполненные ячейки)
5. Результат — набор retention-точек, которые передаются в power law фиттинг

**Валидация:**
- Минимум 2 когорты
- Минимум 3 периода с данными (после усреднения)
- Предупреждение если когорты сильно различаются по размеру (>3x разница между min и max)
- Предупреждение если retention по отдельным когортам аномально отличается от среднего (>2 std)

**Дополнительная визуализация:** при вставке когортной таблицы показать мини-heatmap (классическая когортная тепловая карта) с цветовой кодировкой retention %. Это и полезно для проверки данных, и визуально привлекательно.

**Приоритет:** Must have

---

**Режим 3: Paste DAU + New Users (деконволюция)**

Пользователь вставляет ежедневные агрегированные данные: дату, количество новых пользователей и DAU. Калькулятор восстанавливает retention-кривую математически.

**Формат ввода:** textarea для вставки таблицы (TSV/CSV).

**Ожидаемая структура:**

```
Date        | New Users | DAU
2026-04-01  | 150       | 1200
2026-04-02  | 140       | 1180
2026-04-03  | 160       | 1210
...         | ...       | ...
2026-04-28  | 145       | 1350
```

**Математическая модель (деконволюция):**

DAU в день d складывается из вкладов всех предыдущих когорт:

```
DAU(d) = Σ new_users(i) × R(d - i),  для всех i ≤ d
```

Где R(t) — искомая retention-функция. Задача: найти R(t), которая минимизирует ошибку между наблюдаемым DAU и рассчитанным.

**Алгоритм решения:**
1. Для каждого дня t (от 0 до max) вычисляем R(t) последовательно:
   - R(0) = 1 (по определению)
   - R(1) = (DAU(1) - new_users(1)) / new_users(0) — упрощённо для одной когорты
   - Для общего случая: решаем систему методом наименьших квадратов (non-negative least squares, чтобы R(t) ≥ 0)
2. Сглаживание полученных R(t) — скользящее среднее или isotonic regression (retention не должен расти)
3. Результат — набор точек R(t), которые передаются в power law фиттинг

**Валидация:**
- Минимум 14 дней данных (рекомендовано 28+)
- New Users > 0 для каждого дня (нужен приток для деконволюции)
- DAU ≥ New Users для каждого дня (иначе данные некорректны)
- Предупреждение при высокой волатильности New Users (>50% std/mean) — результат может быть шумным

**Ограничения (показать пользователю):**
- Метод предполагает что retention-функция одинакова для всех когорт (нет сезонности между когортами)
- Точность снижается при малом количестве дней наблюдения
- Не учитывает реактивацию (вернувшиеся пользователи завышают DAU и, как следствие, retention)

**Визуализация:** после деконволюции показать график «Observed DAU vs Reconstructed DAU» — насколько восстановленная модель совпадает с реальными данными. Это и проверка качества, и наглядная демонстрация метода.

**Приоритет:** Should have (сложнее в реализации, требует больше валидации)

### 3.2. Математическая модель

#### Power Law Retention Model

```
R(t) = a × t^(-b)
```

Где:
- `R(t)` — retention в момент времени t
- `a` — коэффициент (начальный уровень retention)
- `b` — показатель затухания (чем больше, тем быстрее падает)
- `t` — номер периода (день, неделя, месяц)

**Фиттинг:** метод наименьших квадратов в логарифмическом пространстве.
Логарифмируем обе стороны: `ln(R) = ln(a) - b × ln(t)` — линейная регрессия.

**Почему power law, а не exponential:**
- Retention в играх и приложениях типично имеет «длинный хвост» — быстрое падение в начале, медленное потом
- Exponential decay (`R = a × e^(-bt)`) предполагает слишком быстрое затухание
- Power law лучше описывает поведение реальных когорт (подтверждено в литературе: Eric Seufert, Andrew Chen)

**Ограничения модели (показать пользователю в tooltip/info):**
- Модель не учитывает сезонность
- Не учитывает реактивацию (возврат ушедших пользователей)
- Точность зависит от количества и разброса входных точек
- Для SaaS с месячной подпиской power law может работать хуже, чем логистическая кривая

#### Расчёт LTV

```
LTV(T) = ARPU × Σ R(t), t = 1..T
```

Cumulative LTV — сумма ARPU × retention за каждый период от 1 до горизонта прогноза.

#### Breakeven

```
Breakeven day = min(t) where LTV(t) ≥ CAC
```

Если LTV не достигает CAC на горизонте прогноза — вывести предупреждение.

#### Метрика качества фиттинга

R² (коэффициент детерминации) — показать пользователю, насколько хорошо модель легла на его данные.
- R² > 0.95 — «Отличный фит»
- R² 0.85-0.95 — «Хороший фит»
- R² < 0.85 — «Данные плохо ложатся на power law, результаты могут быть неточными»

#### Confidence Interval (доверительный интервал)

Прогнозная кривая строится не как одна линия, а как коридор (верхняя и нижняя граница). Ширина коридора зависит от:
- Количества введённых точек (больше точек → уже коридор)
- Удалённости прогноза от последней реальной точки (дальше → шире)

**Реализация:** на основе стандартной ошибки регрессии (SE) в log-пространстве. Верхняя граница = `a × t^(-(b - SE))`, нижняя = `a × t^(-(b + SE))`.

**Визуализация:** полупрозрачная заливка между верхней и нижней границей на графике retention curve. Цвет — тот же что у основной линии, opacity 15-20%.

#### Предупреждение о дальности экстраполяции

Если горизонт прогноза превышает 3× последней введённой реальной точки — показать warning-баннер:

- **Жёлтый (caution):** горизонт > 3× последней точки. Текст: «Прогноз за пределами Day N основан на экстраполяции. Рекомендуется добавить промежуточные retention-точки для повышения точности.»
- **Красный (warning):** горизонт > 10× последней точки. Текст: «Прогноз значительно выходит за пределы имеющихся данных. Результаты носят оценочный характер.»

На графике: вертикальная пунктирная линия на отметке последней реальной точки, визуально разделяющая зону «данные» и зону «прогноз».

#### Industry-Adjusted Mode (режим с учётом индустриальных бенчмарков)

Если пользователь выбрал пресет индустрии И ввёл свои retention-точки — калькулятор может работать в двух режимах (переключатель):

1. **Pure fit** (по умолчанию) — кривая строится только по данным пользователя
2. **Industry-adjusted** — кривая корректируется с учётом формы индустриального бенчмарка

**Логика industry-adjusted:**
- Для каждой введённой точки вычисляем отклонение от бенчмарка: `ratio(t) = user_retention(t) / benchmark_retention(t)`
- Усредняем ratio по всем точкам → получаем `avg_ratio`
- Прогнозная кривая: `R_adjusted(t) = benchmark(t) × avg_ratio`

**Когда полезно:** пользователь ввёл только D1 и D7, но хочет прогноз на D60+. Pure fit по двум точкам ненадёжен, а industry-adjusted использует форму проверенной кривой, скорректированную под уровень конкретного продукта.

**Визуализация:** при включении режима на графике отображаются обе кривые (pure fit — пунктиром, industry-adjusted — сплошной), чтобы пользователь видел разницу и мог выбрать более реалистичный вариант.

#### KPI-карточки (над графиками)

- **Model Fit (R²)** — качество фиттинга

#### Cohort P&L (мини-отчёт по когорте)

Отдельный блок под KPI-карточками. Показывает экономику конкретной когорты в абсолютных деньгах.

**Входные данные:** Cohort size × CAC × ARPU × retention model.

**Содержание:**

| Метрика | Формула | Пример |
|---------|---------|--------|
| Acquisition Cost (total) | Cohort size × CAC | 1000 × $10 = **$10,000** |
| Revenue at Breakeven | ARPU × Σ R(t) × Cohort size, t=1..breakeven | **$10,080** |
| Revenue at Horizon | ARPU × Σ R(t) × Cohort size, t=1..horizon | **$18,400** |
| Profit at Horizon | Revenue at Horizon − Acquisition Cost | **$8,400** |
| ROI at Horizon | Profit / Acquisition Cost × 100% | **84%** |
| Breakeven | День/неделя/месяц когда cumulative revenue ≥ acquisition cost | **Day 47** |

**Визуализация:** под таблицей — компактный waterfall или area chart: зона расходов (flat line = acquisition cost) vs. нарастающий cumulative revenue. Точка пересечения — breakeven. Всё на одном графике, цветовые зоны (красная до breakeven, зелёная после).

**Если CAC не введён:** блок скрыт, показывается hint «Введите CAC чтобы увидеть P&L по когорте».

## 5. Технический стек

### Жёсткие требования

- **Хостинг:** GitHub Pages (статика, без бэкенда)
- **Build output:** статические HTML/CSS/JS, без серверного рендеринга
- **Не зависит от env-переменных** — все данные (`presets.json`, `methodology.md`) комитятся в репо

### Рекомендуемый стек

Учитывая объём фичей (3 режима ввода данных, деконволюция, scenario comparison, industry-adjusted mode, confidence interval, страница методологии с роутингом, парсинг markdown), **vanilla HTML+JS превращается в спагетти**. Поэтому:

- **React 18 + Vite** — сборка в статику, быстрый dev-experience
- **Recharts** — графики (retention curve, LTV, revenue bar, cohort heatmap)
- **Tailwind CSS** — стилизация (dark theme as default)
- **react-router-dom** с `HashRouter` — роутинг на GH Pages
- **react-markdown** + `remark-gfm` — рендер `methodology.md` на странице методологии
- **Vitest** + React Testing Library — юнит-тесты математики и парсеров

### Математика

- Линейная регрессия в log-пространстве для power law fit — ~20 строк, без внешних библиотек
- Non-negative least squares для DAU деконволюции — можно реализовать вручную (Lawson-Hanson) или взять `ml-matrix` / `ml-regression`. Если внешняя библиотека — она должна быть tree-shakeable
- Никаких ML-фреймворков (TF.js, Brain.js) — overkill

### CI/CD

- GitHub Actions: на push в `main` → `npm run build` → деплой `dist/` в branch `gh-pages`
- Lint + типы + тесты обязательны как gating step

### Что осознанно НЕ берём

- TypeScript — для проекта такого размера лишний оверхед, JSDoc на ключевых функциях достаточно. *(Переоценить, если калькулятор будет расти.)*
- Server-side rendering, Next.js — несовместимо с задачей (статика на GH Pages)
- Backend для шеринга — параметры в URL hash покрывают сценарий

## 10. Что осознанно вне скоупа v1

Чтобы калькулятор был «простенький и не black box» — следующие вещи НЕ делаем в первой версии:

- ~~**Месячная шкала** для Subscription Apps~~ — реализовано в v2 (см. `docs/spec-v2-subscription.md`)
- **SaaS B2B** (NRR/GRR/cohort logo retention) — кандидат на v3
- **Реактивация** (resurrected users) — power law их не моделирует
- **Сезонность** — для iGaming/Sportsbook серьёзный фактор, но усложнит UX
- **Multi-product LTV** (cross-sell) — за пределами unit-economics одного продукта
- **Backend для accounts/saved scenarios** — шеринг через URL hash покрывает 80% сценариев
- **A/B симулятор** (тестирование изменений retention) — отдельный инструмент
- **PDF-экспорт всего отчёта** — пока хватит CSV + PNG графика

Эти пункты документируем на странице `/methodology` в секции «Roadmap / лимитации» — чтобы пользователь видел, что инструмент честный про границы.
